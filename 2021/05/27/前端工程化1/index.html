Webpack原理

一. webpack核心流程

1. 什么是webpack

​ webpack是一个现代JS应用程序的静态模块打包器static module bundler。当webpack处理应用程序的时候，他会构建一个依赖关系图dependency graph，其中包含应用程序需要的每个模块，然后将这些模块打包成一个或者多个bundle。
​ webpack就像一条生产线，要经过一系列的处理流程后才能将源文件转换成输出结果呢。这条生产线上的每个处理流程的职责都是单一的，多个流程之间存在依赖关系，只有完成当前处理后才能交给下一个流程去处理。插件就像是插入到是生产线中的一个功能，在特定的时机对生产线上的资源做处理。
​ webpack通过Tapable来组织这条生产线。webpack在运行过程中会广播事件，插件只需要监听他所关心的事件，就能加入到这条生产线中去，去改变生产线的运作。webpack的事件流机制保证了插件的有序性，使得整个系统扩展性良好。

webpack将各种类型的资源，包括图片、css、js等，转译、组合、拼接、生成JS格式的bundler文件。

webpack完成了 模块打包+编译兼容+能力扩展 ：

  • 模块打包：将不同模块的文件打包整合在一起，保证他们之间的正确引用与执行有序
  • 编译兼容：可以利用webpack的loader机制，将一些浏览器无法识别格式的文件比如说.less,.vue,.jsx等，进行编译转换成可以识别。
  • 能力扩展：利用的时webpack的插件机制plugin，在模块打包和编译兼容的基础上，进一步实现比如按需加载、代码压缩等一系列功能，进一步提高自动化程度、工程效率以及打包输出质量

2. webpack核心概念

  • Entry

    起点/入口 指示webpack应该使用哪个模块，来作为其构建内部依赖图的开始。进入入口起点后，webpack会找出哪些模块和库是入口起点直接或间接依赖的。每个依赖项随即被处理，最后输出到bundles的文件中。

  • Output

    output属性告诉webpack在哪里输出它所创建的bundles，以及如何命名这些文件，默认值为./dist。
    基本上，整个应用程序结构，都会被编译到你指定的输出路径的文件夹中。

  • Compiler

    编辑管理器，webpack启动后会创建Complier对象，该对象会一直存活直到结束退出

  • Compilation

    单次编辑过程中的管理器，比如watch=true时，运行过程中只有一个compiler，但每次文件变更触发重新编译时，都会创建一个新的compilation对象。

  • Module

    模块，在webpack中，一切皆模块啊，一个模块对应着一个文件。Webpack会从配置的Entry开始递归找出所有依赖的模块。

  • Chunk

    代码块，一个Chunk由多个模块组合而成，用于代码合并与分割。在编译完成准备输出时，webpack会将module按照特定的规则组织成一个一个的chunk，这些chunk在某种程度上会跟输出一一对应。

  • Loader

    loader能让webpack处理那些非JavaScript文件，因为webpack自身只理解JavaScript。是资源内容转换器。
    loader可以让所有类型的文件转换为webpack能够处理的有效模块，然后利用webpack的打包能力对他们进行处理。本质上，webpack loader将所有类型的文件，转换成应用程序的依赖图（和最终的bundle）可以直接引用的模块。

  • Plugin

    插件。loader被用于转换某些类型的模块，而插件Plugin则可以用于执行范围更广的任务.
    插件的范围包括，从打包优化和压缩，一直到重新定义环境中的变量。插件接口功能极其强大，可以用来处理各种各样的任务。

    3. webpack运行过程

    webpack的运行流程是一个串行的过程，从启动到结束依次执行以下流程：

    初始化阶段：

    1. 初始化参数：从配置文件和Shell语句中读取与合并参数，得出最后的参数。

    2. 开始编译：用上一步得到的参数初始化Complier对象，加载所有配置的插件，执行对象的run方法开始执行编译。

    3. 确定入口：根据配置中的entry找到所有的入口文件。

    4. 编译模块：从入口文件出发，调用所有配置的Loader对模块进行翻译，再找出该模块依赖的所有模块，再递归本步骤直到所有入口依赖的文件都经过了本步骤的处理。

    5. 完成模块编译：在经过上一步使用Loader翻译完所有模块之后，得到了每个模块被翻译后的最终内容以及他们之间的依赖关系。

    6. 输出资源：根据入口和模块之间的依赖关系，组装成一个个包含多个模块的Chunk代码块，再把每个代码块Chunk转换成一个单独的文件加入到输出列表，这一步是可以修改输出内容的最后机会。

    7. 输出完成

      整个过程中webpack会通过发布订阅的模式，向外抛出一些hooks，而webpack插件可以通过监听这些关键的事件节点，执行插件任务而达到干预输出结果的目的。

    打包过程中，compiler对象是一个全局单例，她负责把控整个webpack的打包流程的构建，整个webpack过程中，只有一个compiler对象。

    而compilcation对象是每一次构建的上下文对象，它包含了当次构建所需要的所有信息，每次热更新和重新构建，compiler都会生成一个新的compilation对象，负责此次更新的构建过程。

    每个模块间的依赖关系，则依赖于AST语法树。每个模块文件再通过Loader解析完成后，会通过acorn库生成模块代码的AST语法树，通过语法树就可以分析这个模块是否还有依赖的模块，进而继续循环执行下一个模块的编译解析。

    最终webpack打包出来的bundle文件是一个IIFE的执行函数。

三. sourceMap

sourceMap是一项将编译、打包、压缩后的代码映射回源代码的技术，由于打包压缩后的代码并没有阅读性可言，一旦开发过程中遇到问题，直接在混淆代码中进行debug很困难，是很糟糕的体验，sourceMap可以帮我们快速定位到源代码的位置，提高我们的开发效率。

因为sourceMap是一种源码映射技术，所以有一个映射文件，来标记混淆代码里面对应的源码的位置，这份文件通常以.map结尾。

有了这份映射文件，我们只需要在压缩代码的最末端加上这句注释，就可以让sourceMap生效

有了这段注释之后，浏览器会通过sourceURL去获取这份映射文件，通过解析器解析后，实现源码和混淆代码之间的映射。因此sourceMap也是一项需要浏览器支持的技术。

四. Loader实现原理

loader负责文件转换

webpack最终打包出来的是一份javascript代码，实际上webpack内部也默认只能处理javascript代码。在打包过程中，会默认把遇到的文件都当作javascript代码进行解析。而当项目中存在非js类型的文件时，我们需要先通过Loader进行转换，才能继续执行打包任务。

loader在module.rules中配置，作为模块的解析规则，类型为数组，每一项都是Object

loader的配置

通过loader的配置可以看出，针对每个文件类型，loade是支持以数组的形式配置多个的，因此当webpack在转换该文件类型是，loader会被顺序链式调用，前一个loader的返回的内容会作为下一个loader的入参。因此loader开发会遵循一些规范，比如返回内容必须是标准的js代码字符串，以保证下一个loader能够正常工作，同时在开发时需要严格遵循”单一职责“，只关心loader 的输入以及其对应的输出。

loader函数中的this上下文由webpack提供，可以通过this对象提供的相关属性，获取当前loader需要的各种信息，实际上，这个this指向一个叫做loaderContext的loader-runner特有对象。

五. plugin实现原理

plugin插件负责功能拓展。

webpack是基于发布订阅模式，在运行的生命周期中会广播出许多事件，插件通过监听这些事件，可以在特定的阶段执行自己的插件任务，从而实现自己想要的功能。

compiler和compilation是webpack的两个非常核心的对象，其中compiler暴露了和webpack整个生命周期相关的钩子 compiler-hooks

compilaton暴漏了与模块和依赖相关的粒度更小的事件钩子 complication hooks

webpack的事件机制基于webpack自己实现的一套Tapable事件流方案

plugin在plugins中单独配置，类型为数组，每一项是一个plugin实例，参数都通过构造函数传入。
