Vue vs React

1.核心思想

二者核心思想都是组件式开发，但是：

Vue早期定位是为了降低前端开发的门槛。所以vue推崇灵活易用（渐进式开发体验），数据可变，双向数据绑定（依赖收集）

React是函数式编程（纯组件），数据不可变，单向数据流。函数式编程也就是稳定无副作用以及可测试也就是输入相同输出相同，本质还是函数式编程。

1.1 vue

vue是经典html css js结构，推崇template模板，提供一些自己的指定vue-if vue-show vue-for等。有种 结构 表现 行为 分离的感觉，很像写经典web应用的感觉。
同时，对于组件数据，vue2.0通过Object.defineProperty对数据做到更细致监听，精准实现组件级别更新

1.2 React

React是函数式组件开发，组件采用的jsx语法是把html，css都写入js中去。所有内容都像在写js。
对于组件，当组件调用setState或者props变化时，组件内部render会重新渲染该组件以及这个组件对应的所有子组件，（这点没有vue好），可以通过shouleComponentUpdate或者PureComponent避免不必要的重新渲染。

1.3 核心思想导致写法差异

  • vue推崇template（类似于传统前端html形式，简单易懂）、单文件vue
  • React推崇JSX，Hooks

1.4 核心思想导致api差异

  • 因为vue定位就是简单上手，基于template模板+options API。所以API比较多
  • React本质上就是一个VDOM+diff算法，api少，会setState就能开发了

1.5 核心思想导致社区差异

  • vue定位简易上手，能快速解决问题，所以vue官方主导开发了很多常见问题解决方案，比如vue cli vue router vuex等
  • react只关注底层，上层应用解决方案基本不插手，上层社区最初有redux之类的推出一系列状态管理库进行状态管理。

2. 组件实现不同

vue源码是把options挂载到vue的核心类上，然后通过new Vue({options})拿到实例，也就是vue组件的script导出的是一个挂满options的对象而已，options api中的this指向的是内部的vue实例，对用户是不透明的。

React内部实现比较简单，直接定义了render函数用来生成vnode。React内部使用了四大组件类包装Vnode，不同类型的Vnode使用相应的组件类处理，职责清晰明了。React的this都继承React.Component,this指向用户自定义的类，也就是this对于用户来说是透明的。

3. 响应式原理不同

  • vue双向数据绑定 依赖手机，自动优化，数据是可变的。Vue递归监听data的所有属性，当他们发生变化时，通知每个属性对应的订阅者，订阅者更新。

  • Raect是基于状态机的，数据不可变，是单向数据。需要setState驱动新的State去替换老的State

    当数据改变时，会议组件为根目录，默认这个根下面的所有子元素都进行重新渲染。

4. Diff算法不同

其实vue和react的diff算法核心思想是一致的，都是两个阶段

  1. 不同的组件有不同的DOM结构也就是不同的VDOM。当VDOM type不同时，对应DOM操作就是直接销毁老的DOM节点，创建新的DOM节点。
  2. 同一层次的一组子节点，可以通过唯一的key区分。
  • Vue基于snabbdom库，他有较好的速度以及模块机制。Vue Diff使用双向指针,一边对比一边更新DOM，可以做到精准更新。
  • React采用的是diff队列保存需要更新的DOM，得到patch树，再统一批量操作更新DOM

Vue进行diff时，调用patch打补丁函数，一边比较一边给真实的DOM打补丁
Vue对比节点，当节点元素类型相同，但是className不同时，认为是不同类型的元素，删除重新创建，而react则认为是同类型节点，进行修改操作
① Vue的列表比对，采用从两端到中间的方式，旧集合和新集合两端各存在两个指针，两两进行比较，如果匹配上了就按照新集合去调整旧集合，每次对比结束后，指针向队列中间移动；
②而react则是从左往右依次对比，利用元素的index和标识lastIndex进行比较，如果满足index < lastIndex就移动元素，删除和添加则各自按照规则调整；
③当一个集合把最后一个节点移动到最前面，react会把前面的节点依次向后移动，而Vue只会把最后一个节点放

5. 事件机制不同

  • Vue采用的是原生事件也就是标准web事件
  • Raect采用的是合成事件，通过事件委托的方式，将所有事件都冒泡到document顶层进行监听，然后在这里合成事件下发。也就是因为这个，可以更好的跨端使用事件机制，而不是和web dom强绑定。

补充：关于React Hooks去掉this的使用，this为什么会产生副作用？

因为this是在js语言中，是运行时才能确定的东西。函数式编程思想是不允许这样的，this最主要的副作用是导致组件的生命周期，这和函数式编程{数据=>view}的思想是相悖的，React Hooks弱化了生命周期的概念。（之前类组件也有this，但是再React Class Component中类组件的this是确定的，永远指向当前组件，但是也不好，所以打造出了hooks）

补充：JSX如何转换成DOM？

jsx是js的扩展，具有js的所有能力。

babel是一个工具链，主要是讲ES2015以上的高版本js转换成低版本的js，以便兼容低版本的浏览器或者环境

js=>DOM

  1. babel将jsx转换成es5，也就是jsx=>React.createElement(type,element,value,children) (注意，只是将代码换成这个样子，不是运行这个函数)
  2. 执行这个Raect.createElement函数，函数对形参进行逻辑处理，return一个方法：ReactElement(type,key,ref,self,source,props,ReactCurrentOwner.current)
  3. 执行ReactElement函数，对传进来的这一大堆参数进行重新组装，返回一个element对象，也就是所谓的虚拟DOM vDOM
  4. 最后，这个虚拟DOM在reactDOM.render函数中进行渲染，这样jsx就变成真实dom。（注意，此render并不是组件中写的那个render）

补充：虚拟DOM存在的意义

  • 性能方面，真实dom是直接操纵，会引起大量重绘等。虚拟dom是间接操纵更新dom，但是间接一层本身并不会快，快的是可以在间接虚拟dom上运用diff算法进行合并优化，寻求一个最小变化。
  • 开发爽了 有了虚拟dom，完全颠覆了前端开发方式。只要我们写函数方法，就可以将它渲染成页面，这对模块化组件化开发都有很大的作用
  • 跨平台 这个才是虚拟dom最大的作用，因为虚拟dom本质是一个对象，所以有了虚拟dom可以在不同的平台渲染不同的真实dom，一套代码多段运行，爽！
