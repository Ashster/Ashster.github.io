HTTPS解析

HTTP存在的问题

  • 使用明文传输不加密，内容可以被第三方截获
  • 不验证对方身份，可能遭遇伪装
  • 无法验证报文的完整性，有可能内容已被篡改

HTTPS实质

HTTPS=HTTP+SSL/TLS

更具体得讲，HTTPS就是再HTTP的基础上通过对称加密与非对称加密相结合的方式，加上数字证书认证，来保证传输安全。

传输安全机制包括：

  • 加密处理（防止三方窃听）
  • 认证机制（确认对方身份）
  • 完整性保护（防止传输过程中信息被篡改）

HTTPS并不是一层新的协议，只不过是在HTTP所在的应用层和TCP所在的传输层之间又新加了一层SSL层。

SSL和TLS

SSL在出到第三个大版本时，就是TLS。
也就是说 SSL3.0就是TLS1.0，之后又出了TLS1.1,TLS1.2。当前主流版本还是SSL3.0和TLS1.0。

TLS是以SSL为原型开发的协议，有时会统称为SSL

HTTPS加密原理

https采用对称加密和非对称加密相结合的方式进行加密

对称加密

客户端和服务端共享一个密钥，加密和解密使用的是同一个密钥。

存在问题：以共享的方式分享密钥，必须要将密钥发送给对方，这个过程如果被第三方监听截获，密钥讲失去意义，这种加密方式也不再安全。

非对称加密

有两把密钥，分别为私钥和公钥。公钥加密只能由私钥解开，私钥加密只能由公钥解开。

加密过程:

  • 客户端第一次发请求和服务器协商时，服务器就生成了一对公钥和私钥。
  • 服务器把公钥发送给客户端（明文传输，没有任何加密保护），客户端接收之后，随机生成一个随机数client_random密钥，使用服务器发过来的公钥进行加密
  • 客户端把使用公钥加密的随机数发送给服务器，服务端接收到之后，用配对的私钥进行解密，就获得了客户端生成的那个密钥。
  • 这时，客户端和服务器所持的密钥都是相同的。交换密钥环节就完成了。

之后通信就采用的以上所说的共享密钥进行。

同时使用

其实上面的非对称加密表达的就已经是对称加密与非对称加密相结合的形式了。

为什么要用非对称加密与对称加密相结合，也就是为什么要使用非对称加密，然后再得到相同的密钥，进行共享密钥加密通信呢？

因为公开密钥加密比共享密钥加密方式复杂得多，所以一般采用非对称加密的方式保证密钥共享过程中的密钥的安全性，共享过相同的密钥后，之后交换信息都是使用的对称加密也就是使用相同的密钥进行。

存在问题：因为公钥需要由服务器发送给客户端，而且是明文发送传输，发送过程中公钥可以被三方篡改或者是截获

因此，我们需要一个确认机制，确认客户端收到的公钥就是服务端发过来的，没有被篡改

使用证书保证公钥的正确性

为了防止客户端收到的服务器发送过来的公钥被掉包或者篡改，我们采用数字证书机构和其相关颁发的公开密钥证书。

数字证书认证机构流程(CA)：

数字签名(对数字证书hash和hash算法 用CA机构的私钥进行签名)

  • 首先，CA会向申请者颁发一个数字证书，这个数字证书中有很多内容，比如：签发者、证书用途、使用的HASH算法、签名所使用的算法、证书的到期时间等等
  • 紧接着，把这个数字证书，做一次HASH求值，得到一个HASH值
  • 再接着，利用CA的私钥对这个HASH值和所使用的HASH算法进行加密，这样就完成了数字签名。用CA的密钥加密后，就时独一无二的签名，任何篡改证书的常使，都会被数字签名发现
  • 最后，把数字签名，放在数字证书的末尾，传递给服务器。

接下来，服务器会把这个由数字证书认证机构（CA机构）颁发的公钥证书发送给客户端

  • 客户端拿到这个服务端发来的证书后，用CA机构的公钥进行解密末尾的数字签名，得到hash计算过的数字证书以及hash加密方式
  • 按照这个hash算法对数字证书进行加密，得到数字证书的hash，和刚才解密数字签名中hash加密过的数字证书进行对比，如果一致就认证通过
  • 如果认证通过，就可以取得服务器的公开密钥（数字证书中）

CA公钥：

CA自身的数字证书在我们安装操作系统的时候就已经一起安装了，这样，CA就可以通过自己的私钥对发布的数字证书进行签名，而再客户端就能够用对应的公钥来进行解密

这里就用到的时非对称加密算法，只不过这个加密算法用来签名而不是加密。使用私钥加密，公钥解密，适用于公钥的持有者验证通过私钥加密的内容是否被篡改，但是不保证内容是否被他们截获。使用公钥加密，私钥解密，是相反的，他不保证信息被他人篡改，但是保证信息无法被中间人获得。

HTTPS的建立过程：

  • 客户端发送Client Hello报文开始SSL通信。报文中包含了客户端支持的SSL的指定版本、加密组件（Cipher Suite）列表，加密组件列表就是所使用的加密算法以及密钥长度等。同时会发送一个随机数client-random。
  • 服务器进行SSL检查，以 Server Hello报文作为应答。和客户端一样，在报文中包含SSL版本以及加密组件。服务器返回的加密组件内容是从接收到的客户端加密组件中筛选出来的。同时会发送一个随机数server-random。
  • 服务器发送证书报文。报文中为公开密钥证书。
  • 服务器发送Server Hello Done报文通知客户端，最初阶段的SSL握手协商部分结束
  • SSL握手结束之后，客户端发送Client Key Exchange报文作文回应。报文中包含通信加密中使用的被称为Pre-master secret的随机密码串。这个报文使用上一步的解析的服务器的公钥进行加密。
  • 借着客户端发送Change Cipher Spec报文。这个报文会提示服务器，在此报文之前的通信会采用pre-master secret密钥加密
  • 客户端发送finished报文。该报文连接至今全部报文的整体校验值。这次握手协商能否成功，要以服务器是否能够正确解密该报文作为判断。
  • 服务器同样发送Change Cipher Spec报文
  • 服务器同样发送Finished报文
  • 服务器和客户端的finished报文交换完毕之后，SSL链接就算建立完成了。当然，通信会受到SSL的保护。从此处开始进行应用层协议的通信，即发送HTTP请求
  • 应用层协议通信，发送HTTP请求
  • 客户端断开连接。断开连接时，发送close_notify报文。这部之后再TCP挥手断开连接。

另外，在以上流程图中，应用层发送数据时会附加一MAC的保温摘要，MAC能够查知报文是否遭到篡改，从而保证报文的完整性。
